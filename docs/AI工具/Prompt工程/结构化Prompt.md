# 结构化 Prompt 工程

结构化 Prompt 用**层级结构**和**属性词**组织内容，提升可读性和模型表现，已被网易、字节等大厂在生产中使用。

## 1. 结构化 vs 随意写法

| 对比 | 随意写法 | 结构化写法 |
|------|----------|------------|
| 格式 | 一大段文字 | Markdown/YAML/JSON 层级 |
| 可读性 | 难维护 | 层次清晰、易迭代 |
| 性能 | 易被忽略细节 | 更贴合模型训练习惯 |

## 2. 常用属性词

| 属性 | 含义 |
|------|------|
| **Role** | 角色身份与职责边界 |
| **Profile** | 作者、版本、语言、描述等元信息 |
| **Rules** | 行为规则与约束 |
| **Workflow** | 工作流程步骤 |
| **Initialization** | 初始化与欢迎语 |

## 3. 生产级通用框架

```
ROLE/SCOPE        # 角色与边界
GOAL              # 目标
CONTEXT           # 上下文（用分隔符与数据隔离）
INPUT             # 输入格式
CONSTRAINTS       # 约束
OUTPUT FORMAT     # 输出格式
QUALITY BAR       # 质量标准 / 检查项
```

**安全**：外部内容用 `<untrusted>` 标记，防止提示词注入；指令与数据严格隔离。

## 4. 进阶技巧

- **Chain of Thought（分步思考）**：让模型按步骤分析
- **Few-Shot（示例学习）**：给 1–3 个示例定格式和风格
- **约束条件**：明确技术栈、输出格式、字段限制
- **对比分析法**：用表格对比多方案优劣
- **错误预演**：提前列出可能问题与应对

## 5. 核心原则

1. 先建立**评估体系**再调优
2. 能结构化就结构化（优先 Schema、Structured Outputs）
3. 对结论类任务：先输出**证据/解释**，再给结论，降低事后合理化
4. 指令与数据严格分离
