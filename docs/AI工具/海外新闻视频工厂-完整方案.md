# 海外新闻视频工厂 - 完整方案

## 1. 文档用途

本文件用于指导开发智能体或工程团队实现一个“海外新闻自动化短视频生产系统”的完整版本。

系统目标：

1. 自动抓取海外新闻与公开素材
2. 自动生成适合抖音/视频号的中文短视频脚本
3. 自动生成配音、字幕时间轴和视频
4. 后期由用户通过自然语言进行运营控制
5. 前台可接入 QClaw，后台由可控代码系统承担重任务

本文件描述的是长期可运营版本，不是最小可用版本。MVP 请看配套文档《海外新闻视频工厂-MVP方案.md》。

---

## 2. 总体目标

### 2.1 用户体验目标

系统搭建完成后，运营者应尽量只通过自然语言控制系统，例如：

- 给我今天 10 条适合抖音的海外科技新闻
- 做第 3 条，35 秒，偏快讯
- 开头更抓人一点
- 不要人物图，换官网图
- 开始渲染
- 生成抖音标题、简介和评论区引导

系统应自动完成：

- 新闻抓取
- 正文抽取
- 候选选题排序
- 脚本生成
- 图片/网页截图/可用视频素材拉取
- 配音
- 字幕与时间轴对齐
- 视频渲染
- 产物回传和审核

### 2.2 内容目标

优先服务以下内容类型：

1. 海外科技快讯
2. 海外商业新闻解读
3. 海外趋势/盘点类短视频

### 2.3 质量目标

完整版本的视频至少达到以下标准：

1. 前 3 秒有强钩子
2. 字幕与配音严格同步
3. 来源信息可追溯
4. 风格统一，有品牌感
5. 画面不是廉价 PPT，也不是杂乱拼接
6. TTS 自然，节奏适合短视频
7. 素材结构有逻辑，不只是堆图
8. 30-60 秒内可完整传达一条新闻的核心信息

## 3. 架构原则

### 3.1 分层原则

系统分为两层：

1. 前台自然语言运营层
2. 后台视频生产内核

建议结构：

- QClaw：承担自然语言交互、任务解释、运营控制
- Python 后端：承担采集、素材、脚本、配音、字幕、渲染、状态管理

### 3.2 为什么不把所有逻辑都放在 Agent 里

以下任务不适合主要依赖 Agent 自由发挥：

- 批量抓取新闻
- 下载图片/视频文件
- 网页截图
- 任务重试
- 大文件渲染
- 资源归档
- 成本敏感流程

这些能力必须由可控代码系统承担，Agent 只负责“理解”和“调度”。

### 3.3 为什么不建议把完整生产链条建立在 QClaw/OpenClaw 上

QClaw/OpenClaw 非常适合：

- 自然语言交互
- 选题解释
- 文案重写
- 运营控制

但不适合独立承担：

- 长时间批处理
- 稳定爬虫
- 素材二进制文件处理
- 视频渲染编排
- 可观测性和容错要求高的任务

因此推荐：

> QClaw 做前台，自建后端做生产核心

## 5. 模块拆分

## 5.1 自然语言运营控制层

### 目标

让用户只通过自然语言操作系统。

### 输入示例

- 给我今天适合抖音的美国科技新闻
- 做第 2 条，35 秒，快讯风格
- 开头再炸一点
- 只保留官方来源素材
- 重新渲染

### 输出要求

- 候选新闻列表
- 生成脚本
- 预览链接
- 成片链接
- 标题与简介建议

### 技术建议

- 完整版优先接入 QClaw
- 同时保留一个 API 或 Web 面板作为备用控制入口

---

## 5.2 任务编排与状态层

### 目标

承担生产流程的确定性编排。

### 技术建议

- FastAPI
- PostgreSQL
- Redis
- Celery / RQ / Arq

### 职责

1. 创建任务
2. 调度模块
3. 记录状态
4. 重试失败任务
5. 聚合结果
6. 提供查询接口

### 核心状态枚举建议

- created
- fetching_news
- extracting_content
- scoring_candidate
- generating_script
- collecting_assets
- generating_audio
- building_timeline
- rendering_video
- ready_for_review
- approved
- failed

---

## 5.3 新闻采集层

### 目标

稳定抓取海外新闻候选。

### 建议信源

一级信源：

- Reuters
- AP
- BBC
- CNBC
- TechCrunch
- The Verge
- Wired
- 官方 press release

二级线索源：

- Google News RSS
- GDELT
- Hacker News
- Reddit 热帖
- X 热门讨论

### 输出字段

- title
- url
- source
- published_at
- language
- summary_raw
- content_raw
- hero_image
- tags

### 开发要求

1. 支持定时抓取
2. 支持手动触发
3. URL 去重
4. 标题相似度去重
5. 失败重试

---

## 5.4 正文抽取与标准化层

### 目标

将新闻网页转成可用的结构化数据。

### 技术建议

- trafilatura
- readability-lxml
- newspaper3k
- Playwright 作为复杂站点回退方案

### 输出字段

- cleaned_content
- short_summary
- hero_image
- source
- published_at
- page_title
- page_screenshot

### 开发要求

1. 优先用轻量正文抽取
2. 抽取失败时使用浏览器回退
3. 同时产出网页截图
4. 保留原始 URL 与原始页面快照

---

## 5.5 候选选题评分层

### 目标

决定“哪些新闻值得做短视频”。

### 评分维度建议

- 热度分
- 新鲜度分
- 可讲性分
- 素材可得性分
- 风险分
- 平台适配分

### 推荐输出

- 推荐时长：30 / 45 / 60 秒
- 推荐形式：快讯 / 解读 / 盘点
- 推荐角度：例如“对普通用户的影响”“大公司动作”“行业趋势”

### 开发要求

1. 先规则评分，再用模型做最终排序
2. 必须可解释，不能只输出黑盒分数

---

## 5.6 内容生成层

### 目标

把新闻改写成可播脚本。

### 输出内容

- 中文脚本
- 3 个标题
- 2-3 个封面文案
- 评论区引导句
- 分镜草稿
- 关键词高亮建议

### 支持模式

1. 快讯模式（20-35 秒）
2. 解读模式（35-50 秒）
3. 盘点模式（45-60 秒）

### 质量要求

1. 去除翻译腔
2. 尽量口语化
3. 保留核心事实
4. 避免观点失真
5. 适配短视频节奏

### 提示词设计要求

至少维护：

- 初稿 prompt
- 平台化改写 prompt
- 爆点增强 prompt
- 风险收敛 prompt

---

## 5.7 素材抓取层

### 目标

为视频准备可用画面。

### 素材优先级

一级：

- 原文首图
- 官方网站图片
- 产品图
- 公司 logo
- 原文网页截图
- 地图 / 数据图 / 信息卡

二级：

- 可商用图片库
- 可商用视频库

三级（谨慎）：

- 外媒公开视频片段
- 社交媒体公开视频片段

### 开发要求

1. 每个素材必须记录来源
2. 图片下载失败时有截图回退
3. 不鼓励大规模搬运外媒整段视频

---

## 5.8 素材处理层

### 目标

把原始素材标准化成可进模板的资产。

### 功能

- 裁剪比例
- 模糊背景生成
- 压缩
- 统一命名
- 缩略图
- 视觉卡片生成
- 来源角标叠加

### 要求

所有素材都要落存并带 metadata。

---

## 5.9 音频与字幕层

### 目标

生成自然旁白，并提供字幕时间轴。

### 技术建议

- TTS：MiniMax / ElevenLabs / Azure TTS
- 对齐：优先用 TTS alignment，回退 Whisper

### 输出内容

- 音频文件
- duration
- subtitles timeline
- keyword emphasis timeline

### 开发要求

1. 支持固定主播音色
2. 支持语速调参
3. 支持失败重试
4. 结果写库

---

## 5.10 视频渲染层

### 目标

生成具有统一品牌感的高质量短视频。

### 核心技术

- Remotion 主渲染
- FFmpeg 辅助处理

### 模板建议

1. 海外快讯模板
2. 海外解读模板
3. 每周盘点模板

### 基本结构

快讯模板建议：

1. 开头钩子卡
2. 来源确认卡
3. 主事件画面
4. 背景补充卡
5. 影响/提问收尾卡

### 开发要求

1. 模板数据驱动
2. 字幕按时间轴精确对齐
3. 所有字体本地显式加载
4. 支持低清预览与正式导出

---

## 5.11 审核与回传层

### 目标

让运营者在最终发布前快速审核。

### 回传内容

- 低清预览
- 正式成片
- 封面图
- 标题建议
- 简介建议
- 标签建议
- 素材来源摘要

### 可用命令

- 通过
- 换图
- 开头重写
- 标题重做
- 重新渲染

---

## 5.12 复盘分析层

### 目标

为后续选题和模板优化提供数据。

### 建议记录

- 哪类新闻播放高
- 哪种开头完播高
- 哪种标题点击高
- 哪套模板表现好
- 哪类素材组合表现更好

## 10. 风险与边界

### 10.1 边界

本系统目标是“高质量信息型短视频”，不是：

- 电影级叙事视频系统
- 大规模影视剪辑系统
- 全自动零维护系统

### 10.2 风险点

1. 新闻源网页结构变化
2. 素材站版权策略变化
3. TTS 质量波动
4. 模板审美老化
5. 模型成本上升
6. 平台风格变化

### 10.3 应对思路

1. 所有抓取逻辑可替换
2. 所有 TTS/模型供应商可替换
3. 模板数据结构稳定，视觉可迭代
4. 所有关键步骤保留回退机制

---

## 11. 估算的开发量级

如果实现本完整方案，建议按有效开发工作量评估，而不是只按日历时间估算。

大致量级：

- 完整长期可运营版本：160 - 260 有效开发小时

此量级包含：

- 后端骨架
- 数据层
- 新闻抓取
- 素材抓取
- TTS / 时间轴
- Remotion 模板
- QClaw 接入
- 审核回传
- 初步复盘

---

## 12. 最终建议

建议实施策略：

1. 架构按完整方案设计
2. 功能按 MVP 先落地
3. 核心代码系统先稳定
4. QClaw 作为运营入口后接入或同步接入

完整方案的核心思想不是“所有事情都交给 Agent”，而是：

> 用工程化系统承担生产，用自然语言系统承担运营。
