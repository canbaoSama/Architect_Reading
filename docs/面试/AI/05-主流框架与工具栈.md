# 主流框架与工具栈（2025-2026）

> **重点**：OpenClaw 为核心，技术发展趋势为延伸；LangChain、LangGraph、n8n 等为补充。

---

## 一、OpenClaw（重点掌握）

> 当前最火的个人 AI Agent 开源项目，GitHub 30 万+ star，面试高频考点，建议作为首要学习对象。

### 1.1 学习点清单

| 序号 | 学习点 | 掌握程度 |
|:----:|--------|----------|
| 1 | OpenClaw 的定位、核心特性、与传统框架差异 | 能清晰表述并举例 |
| 2 | 文件系统架构：AGENTS.md、SOUL.md、TOOLS.md、MEMORY.md | 理解每个文件的作用与编写方式 |
| 3 | Gateway 与执行引擎的职责划分 | 能画出架构图并说明数据流 |

| 5 | 多 Agent 配置与协作（转派、群聊礼仪） | 能设计简单多 Agent 场景 |

### 1.3 详细面试问答

#### Q1：OpenClaw 是什么？和 LangChain、传统 Agent 框架有什么区别？

**答**：OpenClaw 是**本地优先的 AI Agent 指挥官**，用「文件系统」而非纯代码来定义 Agent 行为，强调个人使用、多通道、模型无关。

**核心区别**：

| 维度 | OpenClaw | LangChain / LangGraph |
|------|----------|------------------------|
| **配置方式** | 文件驱动（AGENTS.md 等），非工程师可参与 | 代码驱动，需编程 |
| **定位** | 个人助手、多通道 Agent 框架 | LLM 应用开发组件库 / 工作流编排 |
| **使用场景** | 日常任务、消息处理、多平台统一入口 | RAG、复杂工作流、企业级集成 |
| **记忆与 RAG** | 内置 MEMORY.md、记忆热插拔 | 需自行集成向量库等 |

两者可配合：OpenClaw 负责「谁来处理、怎么协作」；LangChain 可充当其内部工具链的一环。

---

#### Q2：AGENTS.md、SOUL.md、TOOLS.md、MEMORY.md 分别写什么？

**答**：

| 文件 | 作用 | 典型内容 |
|------|------|----------|
| **AGENTS.md** | 工作方式、协作规则、记忆策略 | 会话读取规则（如最近 50 条）、写入规则、心跳任务（每日/每周）、与其他 Agent 转派逻辑（技术→Coder、支持→Support） |
| **SOUL.md** | Agent 的「人设」与风格 | 性格、语气、回复风格、禁止事项 |
| **TOOLS.md** | 工具白名单/黑名单 | 允许或禁止调用的工具列表，安全边界 |
| **MEMORY.md** | 长期知识库 | 项目决策、常见问题、规范、用户偏好；支持 RAG 集成 |

这些文件在会话初始化时注入到上下文，相当于 Agent 的「操作系统配置」。

---

#### Q3：OpenClaw 的 Gateway 和执行引擎分别做什么？

**答**：

- **Gateway（指挥部）**：消息路由、多通道接入（Telegram、WhatsApp、iMessage）、策略执行、权限控制。决定「这条消息谁来处理、转发给谁」。
- **执行引擎**：嵌入式运行，集成工具管道和安全沙箱，负责「调用模型、执行工具、返回结果」。

**类比**：Gateway 像「客服总台」，执行引擎像「各个工位」。控制与执行分离，便于扩展和安全管控。

---

#### Q4：OpenClaw 如何解决 Agent「断片」、上下文丢失？

**答**：

1. **会话管理**：读取最近 N 条消息，优先重要消息，过滤无关系统消息。
2. **记忆分层**：短期（当前会话）、长期（项目决策、规范、用户偏好）。
3. **RAG 热插拔**：MEMORY.md 或外部 RAG 注入知识；2026 版本支持「记忆热插拔」，可动态挂载知识库。
4. **心跳任务**：定时执行（如每日检查 PR、每周生成周报），保持对项目状态的感知。

---

#### Q5：多模型支持怎么配置？自动回退策略是什么？

**答**：

- **多模型**：通过 `openclaw models add` 添加 Claude、GPT、DeepSeek、Kimi、Ollama 等，可设主模型和备用模型。
- **自动回退**：主模型不可用（超时、限流、错误）时，按配置顺序切换到备用模型；支持 `sequential`（按顺序尝试）、`load-balanced`（负载均衡）、`random`（随机）等策略。
- **成本优化**：可按任务复杂度选模型，如简单任务用 DeepSeek，复杂推理用 Claude。

---

#### Q6：你用过 OpenClaw 做过什么？遇到过什么问题？

**答**（示例，按自己经历调整）：

- **做过**：本地部署，接入 DeepSeek/MiniMax，做日常任务助手；配置 AGENTS.md 定义心跳任务和转派规则；用 MEMORY.md 存项目规范。
- **问题**：① 上下文变长响应变慢 → 压缩、总结、RAG 只注入关键信息；② 工具调用偶败 → 加重试、降级提示；③ 多 Agent 转派不准确 → 优化 AGENTS.md 中的转派描述。

## 二、AI 技术发展趋势（往后技术发展）

> 面试常问「你对 AI 未来怎么看」「Agent 会往哪个方向演进」，可结合以下趋势作答。

### 2.1 核心趋势概览

| 趋势 | 说明 | 面试可答要点 |
|------|------|--------------|
| **从辅助到自主执行** | 2026 年被视为 Agent 元年，从「聊天框」向「自主执行」转变 | Agent 从回答问题升级为自主规划、调用工具、完成任务 |
| **多模态融合** | 文字、语音、图像、视频、环境、情绪全维度感知 | 多模态任务完成效率比纯文本提升约 60% |
| **多智能体系统（MAS）** | MCP、A2A 协议成为 Agent 时代的「TCP/IP」 | 单一 Agent 向多 Agent 协同执行演进，层级式、平等式架构成熟 |
| **具身智能规模化** | AI 与机器人深度结合，进入真实物理场景 | 人形机器人进入初步商业化，从实验室走向工业 |
| **自主学习与进化** | 通过 RL、知识图谱自动更新「越用越聪明」 | 自动记录高频问题、更新知识库、同步行业政策 |

### 2.2 详细面试问答

#### Q1：你认为 AI Agent 未来 1–2 年会怎么发展？

**答**（按以上趋势组织）：

1. **自主执行**：从「我问你答」到「你帮我完成」，Agent 会主动规划、调用工具、闭环执行。
2. **多智能体协作**：MCP、A2A 等协议落地，异构 Agent 高效互通，形成「Agent 互联网」。
3. **多模态**：语音、图像、视频统一接入，客服、办公、创作等场景体验大幅提升。
4. **行业定制化**：从通用工具向行业专家型转变，金融、医疗、法律等垂域深化。
5. **轻量化部署**：中小企业低成本接入，预制场景模块快速落地。

---

#### Q2：MCP 和 A2A 对未来 Agent 生态有什么意义？

**答**：

- **MCP**：统一 AI 应用与外部工具/数据的对接，类似「USB-C」；工具写一次，多处复用。
- **A2A（Agent-to-Agent）**：定义智能体间通信协议，支撑多 Agent 协作、任务转派、能力组合。
- **意义**：形成开放生态，避免各厂商重复造轮子；中小企业也能接入标准能力，加速 Agent 普及。

---

#### Q3：多模态和具身智能对前端/AI 应用开发有什么影响？

**答**：

- **多模态**：前端需支持语音输入、图片上传、视频理解；交互从「文本框」扩展到「全模态输入」。
- **具身智能**：偏硬件和机器人，应用层更多是「与机器人/环境的指令与控制界面」；前端可关注「控制面板、状态展示、异常处理」。
- **通用**：流式输出、状态管理、错误恢复等前端能力依然通用，只是输入输出形态更丰富。

## 三、补充：LangChain、LangGraph、n8n

> 作为补充了解，面试时能说清定位和选型即可；重点仍为 OpenClaw。

### 3.1 LangChain 与 LangGraph

**定位**：LangChain 是 LLM 应用开发组件库；LangGraph 是其高级编排层，支持图结构、分支、循环。

| 对比 | LangChain | LangGraph |
|------|-----------|-----------|
| 模式 | 链式、线性 | 有向图、DAG |
| 适用 | 单轮问答、翻译、简单 RAG | Agent、多轮对话、复杂工作流 |
| 核心 | LCEL（`prompt \| llm \| parser`） | State、Node、Edge、Checkpoint |

**选型**：流程固定用 LangChain；要分支、循环、多 Agent 用 LangGraph。

---

### 3.2 n8n

**定位**：开源工作流自动化平台，支持 AI/LLM 节点，可拖拽搭建 AI Agent 工作流，无需或少量代码。

#### 学习点清单

| 序号 | 学习点 | 掌握程度 |
|:----:|--------|----------|
| 1 | n8n 的定位与适用场景 | 能说清与 Dify、LangChain 的差异 |
| 2 | AI Agent 节点：Brain、Memory、Tools、System Prompt | 知道四要素作用 |
| 3 | 支持的 LLM：OpenAI、Claude、Ollama、Gemini 等 | 了解常用 Provider |
| 4 | 记忆类型：Vector Store、Buffer、Window | 知道如何选 |
| 5 | 典型应用：客户线索筛选、内容起草、RAG 问答 | 能举例 |

#### 详细面试问答

**Q：n8n 是什么？和 Dify、LangChain 比有什么区别？**

**答**：

- **n8n**：开源工作流自动化，偏「自动化 + AI」；节点丰富（HTTP、数据库、代码、AI 等），可搭建复杂业务流程；自托管，数据可控。
- **Dify**：偏 LLM 应用平台，拖拽式 RAG、Agent、API 发布，更适合「快速搭 AI 应用」。
- **LangChain**：代码驱动，偏开发者，灵活度高，适合深度定制。

**选型**：要「自动化流程 + AI」用 n8n；要「AI 对话/RAG 应用」用 Dify；要代码级控制用 LangChain。

---

**Q：n8n 的 AI Agent 节点需要哪几个部分组成？**

**答**：

1. **Brain（LLM 连接）**：OpenAI、Claude、Ollama、Gemini 等。
2. **Memory（上下文）**：Vector Store（语义检索）、Buffer（完整对话）、Window（最近 N 条）。
3. **Tools（工具）**：子工作流、HTTP 请求、代码执行、Wikipedia 等。
4. **System Prompt**：定义 Agent 行为、约束、输出格式。

---

**Q：n8n 适合做什么类型的 AI 应用？**

**答**：

- 客户线索筛选与 CRM  enrichment
- 多步研究与信息汇总
- 支持消息分类与路由
- 知识库问答（配合 RAG）
- 内容起草与规则校验

## 四、Cursor Rules 与 Skills

### 学习点清单

| 序号 | 学习点 | 掌握程度 |
|:----:|--------|----------|
| 1 | Rules 与 Skills 的本质区别 | 被动指导 vs 主动能力 |
| 2 | Rules 适用场景与反例 | 知道什么该写、什么不该写 |
| 3 | Skills 的结构（SKILL.md）与调用方式 | 能写一个简单 Skill |
| 4 | Agent 何时自动调用 Skill | 理解上下文触发逻辑 |
| 5 | Rules vs Skills vs Commands 对比 | 能说清三者的分工 |
| 6 | OpenSkills 与生态 | 了解扩展 Agent 能力的方式 |

### 详细面试问答

#### Q1：Cursor 的 Rules 和 Skills 有什么区别？

**答**：

| 维度 | Rules（规则） | Skills（技能） |
|------|---------------|----------------|
| **作用** | 被动指导，持续影响 Agent 行为 | 主动能力，按需执行 |
| **加载** | 每次对话开始时加载，静态 | 根据任务动态决定是否调用 |
| **内容** | 简短、规范、约束（代码风格、禁止修改） | 详细、多步骤工作流、可复用流程 |
| **存储** | `.cursor/rules/` 或项目级配置 | `.cursor/skills/xxx/SKILL.md` |
| **触发** | 始终生效 | 手动 `/skill`、`@skill` 或 Agent 自动判断 |

**记忆**：Rules 引导「怎么做」，Skills 执行「具体任务」。

---

#### Q2：Rules 适合写什么？不适合写什么？

**答**：

**适合**：
- 代码规范和风格（如 Tailwind、单文件组件约定）
- 工具和框架选择
- 构建、测试命令
- 防护规则（如禁止修改某目录）

**不适合**：
- 很长的工作流（应放到 Skills）
- 整份风格指南（用 linter 更好）
- 过多规则导致上下文膨胀、影响响应质量

---

#### Q3：如何创建一个 Skill？Skill 里通常写什么？

**答**：

1. 在 `.cursor/skills/your-skill-name/` 下创建 `SKILL.md`
2. 或在聊天中输入 `/create-skill` 按指引创建

**SKILL.md 典型内容**：
- 技能目标与适用场景
- 输入输出说明
- 多步骤操作流程（如部署、测试、代码审计）
- 领域知识或专用指令

**调用**：`/skill-name` 手动触发；`@skill-name` 作为上下文附加；或由 Agent 根据当前任务自动调用。

---

#### Q4：Agent 怎么知道该调用哪个 Skill？

**答**：

Agent 会根据当前对话内容、用户意图和 Skill 的描述做匹配。SKILL.md 中的目标、适用场景会被用于判断相关性。因此写好「技能描述」很重要，相当于给 Agent 的「检索提示」。

---

#### Q5：MCP 和 Skills 有什么关系？

**答**：

- **MCP**：模型与外部系统通信的**标准协议**，定义 Tools、Prompts、Resources 等能力的暴露方式。
- **Skills**：在 Cursor 场景下，是**具体的能力封装**，可以依赖 MCP 提供的工具，也可以独立实现。

可以理解为：MCP 是「插座标准」，Skills 是「插在插座上的电器」。

## 五、MCP（Model Context Protocol）

### 学习点清单

| 序号 | 学习点 | 掌握程度 |
|:----:|--------|----------|
| 1 | MCP 是什么、解决什么问题 | 能一句话说清 |
| 2 | MCP 的三类能力：Tools、Prompts、Resources | 知道每类的作用 |
| 3 | MCP 的通信方式（JSON-RPC 2.0） | 了解基本架构 |
| 4 | MCP Server 与 Client 的关系 | 理解谁提供能力、谁消费 |
| 5 | 典型 MCP 应用场景 | 能举例 |
| 6 | MCP 与安全（权限、校验） | 知道要注意什么 |

### 详细面试问答

#### Q1：MCP 是什么？为什么需要它？

**答**：

MCP（Model Context Protocol）是 **AI 应用与外部系统集成的开放标准**，有人比喻为「AI 的 USB-C 口」。

**解决的问题**：不同 AI 应用（Cursor、Copilot、Claude 等）要和数据库、API、文件系统、自定义工具对接时，各自实现成本高、不统一。MCP 定义一套协议，让工具提供方写一次 MCP Server，各类 AI 应用都能通过 MCP Client 使用。

---

#### Q2：MCP 提供哪三类能力？

**答**：

| 类型 | 说明 | 示例 |
|------|------|------|
| **Tools** | 模型可调用的函数 | 查数据库、调 API、执行命令 |
| **Prompts** | 预设的模板与工作流 | 代码 Review 模板、需求分析模板 |
| **Resources** | 供模型或用户使用的数据与上下文 | 文档、配置、知识库片段 |

---

#### Q3：MCP Server 和 MCP Client 分别是什么角色？

**答**：

- **MCP Server**：暴露能力的一方，如数据库 MCP Server、文件系统 MCP Server、自定义业务 MCP Server。实现 Tools/Prompts/Resources 的接口。
- **MCP Client**：集成在 AI 应用（如 Cursor、Copilot）中，连接 MCP Server，把能力呈现给模型使用。

模型通过 Client 发现并调用 Server 提供的能力。

---

#### Q4：MCP 有什么安全方面的考量？

**答**：

- **权限**：需要控制哪些工具/资源对模型可见，避免越权访问。
- **校验**：执行 Function Call 前应校验参数、鉴权，不暴露不该调用的接口。
- **数据**：涉及敏感数据的 Resource 要考虑脱敏、访问控制。
- **执行**：Tools 可能执行代码或系统命令，需要沙箱或白名单机制。

## 七、学习路线建议（按优先级）


## 八、高频面试题速记

| 问题 | 一句话回答 |
|------|------------|
| OpenClaw 和 LangChain 啥关系？ | OpenClaw 是个人 AI 助手/多通道 Agent 框架；LangChain 是 LLM 应用开发组件库；两者解决不同层次的问题，可配合使用。 |
| LangChain 和 LangGraph 选哪个？ | 简单固定流程用 LangChain；要分支、循环、多 Agent 用 LangGraph。 |
| Rules 和 Skills 有什么区别？ | Rules 被动约束行为，Skills 主动执行任务；Rules 始终加载，Skills 按需调用。 |
| MCP 解决什么？ | 统一 AI 应用与外部工具/数据的对接协议，避免各应用重复造轮子。 |
| 你更熟悉哪个框架？ | 按实际情况答，建议**优先能讲清 OpenClaw** 的架构和一次实际使用经历；LangGraph、n8n 可作为补充。 |
