# AI 面试准备路线与资源（新手友好版）

## 一、先定目标：你要投哪类岗位

先选一个主方向，不要同时准备所有岗位。

| 方向 | 更适合谁 | 重点能力 |
|------|----------|----------|
| AI 应用工程师（推荐新手主线） | 前端/后端转型、希望尽快做出项目 | API 调用、RAG、Prompt、工具调用、评测与上线 |
| AI 前端工程师 | 有前端经验，想做 AI 交互体验 | 流式输出、对话 UI、多轮状态、可解释交互 |
| 全栈 AI 工程师 | 有一定后端基础，能独立交付 | 前后端联动、服务化、可观测与成本治理 |

建议：`主线 = AI 应用工程师`，`副线 = AI 前端工程师`。

---

## 二、学习路线（12 周可执行）

核心原则：每 2 周都要有“可演示成果”，避免只看不做。

### 阶段 0（第 1 周）：环境和最小闭环

目标：把第一个 AI demo 跑通。

- 安装与熟悉：Node.js / Python / Git / API Key 管理
- 跑通一个最小对话 demo：输入 -> 调用模型 -> 输出
- 记录第一版 README：依赖、启动、限制

验收：
- 能在 3 分钟内演示一个最小 AI 页面或脚本
- 能解释清楚 `system`、`user`、`assistant` 消息结构

### 阶段 1（第 2-4 周）：LLM 基础 + Prompt + 工具调用

目标：具备“能稳定产出正确格式结果”的能力。

- 学习重点：
  - Token、上下文窗口、采样参数（temperature/top_p）
  - Prompt 模板化、few-shot、结构化输出（JSON）
  - Function Calling / Tool Use 与错误处理
- 产出：
  - 一个“结构化输出”小功能（如简历解析、需求摘要）
  - 一个“工具调用”小功能（如天气/搜索/知识检索）

验收：
- 对同一任务能写出可复用 Prompt 模板
- 输出可被程序稳定解析，不靠手工修补

### 阶段 2（第 5-8 周）：RAG 主线

目标：独立做一个可讲述的 RAG 项目。

- 学习重点：
  - 文档清洗、分块策略、向量化、检索与重排
  - RAG 评测：Recall@k、答案忠实度、延迟、成本
  - 权限、引用、失败降级（查不到时如何处理）
- 产出：
  - 项目 1：文档问答（带引用）
  - 项目 2：业务知识助手（有简单评测页）

验收：
- 能解释为什么这么分块、为什么选这个检索策略
- 能给出至少 3 个可量化指标（准确率/延迟/成本）

### 阶段 3（第 9-10 周）：前端 AI 专项与工程化

目标：把“能用”升级为“可上线演示”。

- 学习重点：
  - SSE/WebSocket 流式输出
  - 对话状态管理、中断重试、错误边界
  - 基础可观测：日志、失败率、耗时、Token 成本
- 产出：
  - 可交互聊天界面（流式渲染 + 引用来源 + 失败回退）

验收：
- 网络抖动、模型超时、空结果都能优雅处理
- 可以展示一张简单监控面板或日志统计

### 阶段 4（第 11-12 周）：面试冲刺

目标：完成“项目可讲 + 问题可答 + 场景可设计”。

- 梳理 2 个项目故事：问题、方案、取舍、结果
- 按岗位题库刷题：基础题 + 场景题 + 系统设计题
- 进行 3 次模拟面试：技术深挖、系统设计、行为面

验收：
- 每个项目都能讲清“为什么不是另一个方案”
- 每个项目都能说出关键指标和改进结果

## 三、新手常见误区与修正

1. 误区：先学很多框架，再开始做项目。  
修正：先做 1 个小项目，再按问题补框架。

2. 误区：只看 Prompt 技巧，不做评测。  
修正：至少建立一个小测试集，版本前后对比结果。

3. 误区：把时间花在冷门细节，不练系统设计。  
修正：每周至少练 1 次“口述完整架构”。

4. 误区：简历写“用了 LangChain”，讲不清 trade-off。  
修正：重点写“你为什么选它、带来什么效果”。

## 四、项目路线（从简单到可面试）

### 项目 1：最小 AI 对话应用（1 周）

- 能力点：API 调用、Prompt、流式输出
- 面试可讲点：消息结构、重试策略、限流与错误处理

### 项目 2：RAG 文档问答（2-3 周）

- 能力点：切分、检索、引用、评测
- 面试可讲点：分块策略对效果影响、检索失败怎么降级

### 项目 3：Agent 工具助手（2 周）

- 能力点：工具调用、多步推理、权限控制
- 面试可讲点：什么时候该用 Agent、如何控成本与风险

---

## 五、面试准备清单（投递前自检）

- 能在白板上画出 RAG 完整链路（离线索引 + 在线检索生成）
- 能解释 3 个常见故障排查路径（答案错、延迟高、成本超）
- 能讲清楚至少 2 个真实 trade-off（效果 vs 成本、速度 vs 准确）
- 有 1 份可运行项目 + 1 份简洁技术文档（架构图 + 指标）
- 行为面准备 3 个故事：冲突、失败复盘、跨团队协作

## 七、每周复盘模板（建议固定执行）

1. 这周做了什么可演示功能？  
2. 哪个环节最卡？是模型、检索、工程还是理解？  
3. 本周新增了什么可量化指标？  
4. 下周要完成的最小里程碑是什么？

---

核心建议：新手阶段最有价值的路径是 `小闭环 -> 可量化 -> 可讲述`。先把 1 个 RAG 项目做扎实，再扩展框架广度，面试通过率会明显更高。
