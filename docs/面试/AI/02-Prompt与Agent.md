# Prompt 工程与 AI Agent

## Prompt 工程

### 1. Zero-shot / Few-shot

| 方式 | 说明 |
|------|------|
| **Zero-shot** | 只给指令，不给示例；适合模型已掌握的任务 |
| **Few-shot** | 给 2–5 个示例；适合固定格式、边界模糊、需稳定输出的任务 |

**原则**：先 zero-shot；质量不够再加 few-shot，示例要覆盖边界情况。

### 2. Chain-of-Thought（链式思考）

- 让模型**分步推理**再给出最终结论
- 适合数学、逻辑、多步推理
- 原理：中间步骤为后续生成提供上下文，提升准确率

### 3. 生产级系统 Prompt 设计

- 角色与边界定义
- Few-shot 示例
- 输出格式（JSON、Markdown 等）
- 边界与越界处理（拒绝、澄清）
- 语气、长度等约束
- **把 prompt 当代码**：版本管理、评审、测试后再上线

### 4. 结构化输出与 Function Calling

- **结构化输出**：约束输出为 JSON/schema，便于下游解析
- **Function Calling**：模型返回「调用哪个函数、传入什么参数」，应用再执行并可选回传结果
- 是 Agent「工具调用」的基础

### 5. Prompt Chaining（提示链）

- 把复杂任务拆成多步，每一步单独调用 LLM
- 例：分类 → 实体抽取 → 检索 → 生成回答
- 优点：每步可控、易调试；缺点：多次调用、延迟增加

### 6. Prompt Injection 防护

- 用户输入中可能包含试图覆盖指令的内容
- **多层防护**：
  - 输入清洗、长度限制
  - 用分隔符区分系统指令和用户输入
  - 明确层级（系统指令优先）
  - 校验 Function Call 再执行，不暴露不该调的工具

## AI Agent

### 1. Agent 与 Chain 的区别

| | Chain | Agent |
|--|-------|-------|
| 流程 | 固定：A → B → C | 动态：思考 → 选工具 → 执行 → 观察 → 再思考 |
| 工具 | 预定义调用顺序 | 根据上下文决定何时调用、调用什么 |
| 适用 | 流程清晰、可预见的任务 | 开放式、需规划和探索的任务 |

### 2. ReAct（推理 + 行动）

- **流程**：思考 → 选工具/动作 → 执行 → 观察结果 → 再思考 → 循环直至完成
- 是当前 Agent 的主流范式

### 3. Tool Calling / Function Calling

- 模型返回「调用函数 + 参数」的结构化结果
- 应用执行函数，将结果再传给模型
- Agent 通过这一机制与外部 API、数据库、计算工具交互

### 4. 多 Agent 协作

- 多个 Agent 分工协作（规划、检索、代码、测试等）
- 常见框架：AutoGen、CrewAI、LangGraph
- 流程示例：用户需求 → 主管 Agent 拆任务 → 各 Agent 执行 → 汇总输出

### 5. MCP 与 Skills

- **MCP（Model Context Protocol）**：模型与外部系统/工具之间的统一通信协议
- **Skills**：把常用能力封装成可复用技能（如代码生成、文档总结）

## 常见面试题速答

**Q：如何减少 RAG 幻觉？**  
用 RAG 约束上下文、要求引用来源、输出校验、关键场景加人工复核。

**Q：稀疏 vs 稠密检索如何选？**  
复杂语义理解用稠密；精确词/代号用稀疏；生产常用混合检索。

**Q：Temperature 如何选？**  
结构化/抽取用 0；对话用 0.3–0.7；创意写作可更高。

**Q：Agent 与 Chain 何时用？**  
流程确定用 Chain；开放式、需规划、需工具调用的用 Agent。
