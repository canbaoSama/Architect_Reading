# LLM 基础与 RAG

## LLM 基础

### 1. Transformer 和注意力机制

- **自注意力**：模型同时看句中所有词，学习哪些词彼此相关
- **Q/K/V**：每个词转为 query、key、value 三个向量，通过 query 与 key 的匹配决定关注哪些位置，再组合 value
- **多头注意力**：并行多次 attention，学习不同关系（语法、语义、位置）
- **工程影响**：attention 与序列长度平方相关，长上下文更慢、更贵，部分信息可能「淹没」

### 2. Tokenization（分词）

- **含义**：把文本转为模型可处理的 token（介于字符和词之间）
- **影响**：
  - 分块大小需考虑 embedding 和生成模型的 token 限制
  - 上下文窗口用 token 计，中文/代码每词 token 更多
  - API 按 token 收费，冗余 token 直接增加成本

### 3. Fine-tuning / RAG / Prompt Engineering 区别

| 方式 | 适用场景 |
|------|----------|
| **Prompt Engineering** | 任务在模型能力内，只需调格式、语气、推理风格；成本低、迭代快 |
| **RAG** | 需要查询专用/实时/私有数据；模型不改，运行时注入文档 |
| **Fine-tuning** | 需要改变模型底层行为、领域语言或格式；成本高、需数据与训练 |

**优先顺序**：先 prompt → 缺知识加 RAG → 需改变行为再考虑微调。

### 4. Embedding 和向量

- **含义**：把文本变成数字向量，语义相近的文本向量相近
- **用途**：RAG 的语义检索、聚类、推荐等
- **注意**：存储和检索必须用**同一 embedding 模型**，否则需全部重算

### 5. 上下文窗口

- **含义**：单次请求允许的最大 token 数（输入+输出）
- **"Lost in the middle"**：中间内容召回较差，不要盲目堆长上下文
- **策略**：用 RAG 只注入最相关片段，对长内容先摘要、再分段处理

### 6. Temperature / top_p / top_k

| 参数 | 作用 |
|------|------|
| **Temperature** | 0=确定性强，0.7 左右=有随机性，>1 易混乱 |
| **top_p（核采样）** | 只从累积概率达到 p 的最小 token 集合中采样 |
| **top_k** | 只从概率最高的 k 个 token 中选 |

**建议**：结构化输出/抽取用 `temperature=0`；对话用 0.3–0.7。

### 7. 幻觉（Hallucination）

- **含义**：输出看似合理但事实错误或虚构
- **应对**：引用来源、输出校验、人工复核、RAG 基于检索内容生成、在 prompt 中强调「仅基于给定内容回答」

### 8. LLM 输出评估

- 结构化任务：精确匹配、正则、JSON schema 校验
- 开放生成：LLM-as-judge（另一模型打分）
- RAG：拆分为 retrieval 质量 + faithfulness（是否忠于检索内容）+ answer relevance

## RAG 架构

### 1. RAG 端到端流程

- **Index 阶段**：文档 → 清洗 → 分块 → 向量化 → 存入向量数据库
- **Query 阶段**：问题向量化 → 检索相似块 → 注入 prompt → LLM 生成回答

### 2. RAG vs 纯 LLM

- 可获取**实时、私有、领域**知识
- 降低幻觉，答案可追溯到原文

### 3. 分块（Chunking）

| 方式 | 特点 |
|------|------|
| 固定长度 | 实现简单，可能截断语义单元 |
| 按段落/小节 | 保留上下文，块大小不均 |
| 层级分块 | 多粒度检索，实现更复杂 |

常见 chunk 大小：256–512 tokens，需结合实际文档和问题测试。

### 4. 稀疏与稠密检索

- **稀疏（BM25 等）**：关键词匹配，快，对专业术语/代号友好
- **稠密（Embedding）**：语义相似，理解意图更好
- **混合检索**：结合两者，如用 RRF（Reciprocal Rank Fusion）融合结果

### 5. 向量数据库对比

| 数据库 | 特点 |
|--------|------|
| Pinecone | 托管云服务，零运维，费用较高 |
| Weaviate | 开源，支持混合检索 |
| Chroma | 轻量，适合原型和小规模 |
| pgvector | 基于 Postgres，适合已有 PG 的场景 |

### 6. RAG 评估三要素

1. **Context Relevance**：检索内容是否与问题相关
2. **Faithfulness**：回答是否忠于检索内容（不瞎编）
3. **Answer Relevance**：回答是否真的回答了问题

### 7. 生产化 RAG 常见问题

- 检索质量差 → 调 chunk、embedding、检索策略
- 幻觉多 → 强化 prompt、加校验、用 RAG 控制输出范围
- 延迟高 → 缓存、预取、优化索引和检索 pipeline
- 知识库动态更新 → 增量索引、版本与更新策略
